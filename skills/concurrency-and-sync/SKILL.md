当学生询问锁、信号量、条件变量、死锁、竞态条件、原子操作、OCC/MVCC 等并发相关问题时，先让学生说出自己的理解，再按以下框架引导。

## 第一步：诊断并发类型（S³ 框架）

Lampson 提出：并发问题先问自己 **Shard, Stream, or Struggle？**

| 类型 | 描述 | OS 典型场景 |
|------|------|------------|
| **Really Easy（Shard）** | 各线程操作不同数据，无共享状态 | 多核各自处理不同进程的地址空间 |
| **Really Easy（Stream）** | 流水线：A 产出，B 消费，通过队列通信 | 生产者-消费者（管道、消息队列） |
| **Easy（Locking）** | 有共享状态，用锁保护 | 内核全局数据结构、文件引用计数 |
| **Nuisance** | 系统允许并发但应用不想处理（如文件重命名冲突） | 目录项的 TOCTOU 问题 |
| **Hard（Serializable）** | 任意共享状态，需要可串行化保证 | 数据库事务，分布式共识 |
| **Eventual** | 所有更新可交换（commute），允许读取过时数据 | DNS 缓存，DNS 更新传播 |

**提问学生**：「你的问题里，线程之间有没有共享状态？如果有，它们的操作能否交换顺序？」

## 加锁的正确方式（"Easy" 并发）

**基本规则**：
- 对任何共享变量的读写，都要持有对应的锁
- 锁保护的是「不可分割的复合操作」，不只是单条语句
- 两个操作「冲突」当且仅当它们不可交换（写-写、读-写均冲突）

**Lock Invariant（锁不变量）**：
- 在任何 unlock 时刻，共享状态必须满足某个不变量
- 示例：`balance >= 0` 在每次 unlock 时都成立
- 提问：「你的锁不变量是什么？释放锁时它还成立吗？」

**死锁预防**：
- 死锁四个条件：互斥持有、持有等待、不可抢占、循环等待
- **唯一可靠预防法**：对所有锁建立全局编号，只按从小到大的顺序获取
- 提问：「画出线程获取锁的顺序图，有没有环？」

## 条件变量的正确使用

```c
// 正确：必须用 while，不能用 if
while (!condition) {
    pthread_cond_wait(&cv, &mutex);
}
// 原因：wait 返回时条件不一定还成立（虚假唤醒 / 被其他线程抢先修改）
```

典型 OS 场景：等待 I/O 完成、等待内存可用（OOM killer）、等待子进程退出（waitpid）

## OCC 与 MVCC（进阶）

- **OCC（乐观并发控制）**：写入本地缓冲区，提交时检查读集合是否被修改
  - 冲突则 abort + retry；适合冲突少的场景
- **MVCC（多版本并发控制）**：每次原子写保存新版本，读取用时间戳选择版本
  - 读不阻塞写；适合读多写少、需要快照语义的场景（PostgreSQL）

## 等待自由（Wait-free）与无锁（Lock-free）

- **Wait-free**：每个线程在有限步骤内完成操作，无论其他线程做什么
- **Lock-free**：至少有一个线程总能前进（不会整体卡死）
- 适用于：高性能队列、计数器（用 CAS/FAA 指令）
- 提问：「你的算法里，一个线程如果挂起，会不会让其他线程永远等待？」

## 常见学生误区

| 误区 | 正确理解 |
|------|---------|
| `i++` 是原子的 | 不是，需要 mutex 或 `__atomic_fetch_add` |
| 只保护写操作就够了 | 读也要保护（data race = 未定义行为） |
| 用 `if` 等待条件变量 | 必须用 `while` |
| 加了锁就不会 deadlock | 多个锁时仍需保证获取顺序 |
| volatile 可以代替锁 | volatile 只防止编译器优化，不保证内存可见性或原子性 |
