当学生询问虚拟内存、分页/分段、地址翻译、页面替换算法、写时复制、内存映射（mmap）等问题时，先让学生说出自己的理解，再按以下框架引导。

## 间接层（Indirection）是 OS 的核心工具

Lampson：「任何 OS 问题都可以通过增加一层间接层来解决。」

虚拟内存就是最典型的间接层：`虚拟地址 → 物理地址`

间接层的五种用途：
| 用途 | OS 体现 |
|------|---------|
| **Multiplex（多路复用）** | 多个进程共享同一物理内存 |
| **Virtualize（虚拟化）** | 每个进程看到独立的地址空间（比物理内存还大） |
| **Interpose（拦截）** | OS 拦截所有内存访问，实施保护检查 |
| **Encapsulate（封装）** | 进程不知道物理内存布局，只看到虚拟地址 |
| **Name（命名）** | 文件名→inode→磁盘块；符号→虚拟地址 |

提问：「为什么进程不直接使用物理地址？间接层解决了哪些问题？带来了什么开销？」

## 分页机制详解

**页表结构**：
- 虚拟地址 = VPN（虚拟页号）+ 页内偏移
- 物理地址 = PPN（物理页号）+ 页内偏移（偏移不变）
- 页表项（PTE）包含：PPN、Valid/Dirty/Accessed/Writable/User 位

**多级页表**：x86-64 用 4 级页表（PML4→PDPT→PD→PT），每级 512 项
- 优点：稀疏地址空间无需分配中间级页表
- 提问：「64 位地址空间如果用单级页表，需要多大？」

**地址翻译流程**：
1. 查 TLB（命中→直接得到 PPN）
2. TLB miss → 页表遍历（x86 硬件 / RISC 软件）
3. PTE.Valid=0 → **Page Fault**（缺页中断）→ OS 处理

## 缺页处理（Page Fault）

OS 收到缺页中断后：
1. 检查是否合法访问（VMA 范围、权限）→ 非法则 SIGSEGV
2. 找一个物理页框（可能需要先换出一个页）
3. 从磁盘加载页内容（若是 demand paging）
4. 更新 PTE，重新执行缺页指令

**关键场景**：
- **Demand Paging**：fork() 后不立即复制，等到真正访问才加载
- **Copy-on-Write（写时复制）**：fork() 后父子共享页面，写时触发缺页并复制
- **Memory-Mapped File（mmap）**：文件内容映射到地址空间，读写文件 = 读写内存

提问：「fork() + exec() 的组合中，写时复制节省了什么？exec() 之后 fork 复制的那些页面还需要吗？」

## 页面置换算法

当物理内存满时，OS 需要选择哪个页面换出：

| 算法 | 原理 | 特点 |
|------|------|------|
| OPT（最优） | 换出未来最晚访问的 | 理论最优，不可实现 |
| LRU | 换出最久未使用的 | 近似最优，实现昂贵 |
| Clock | 环形 + Accessed 位近似 LRU | Linux 实际使用 |
| FIFO | 换出最早加载的 | 简单，可能触发 Bélády 异常 |

**Bélády 异常**：FIFO 中增加物理内存反而导致更多缺页（LRU 不会出现此问题）

提问：「为什么 Linux 不直接实现 LRU？Clock 算法和 LRU 的差异在哪种访问模式下最明显？」

## 内存保护与隔离

- 每个进程有独立的页表 → 天然隔离
- PTE 权限位：U（用户态可访问）、W（可写）、X（可执行）
- **内核映射**：Linux 将内核代码/数据映射到每个进程的高地址区域（高半段）但标记 U=0
- **KASLR/SMEP/SMAP**：现代 CPU 防止内核漏洞利用的保护机制

提问：「为什么内核代码要映射到每个进程的地址空间？不映射会怎样？」

## 常见学生误区

| 误区 | 正确理解 |
|------|---------|
| 虚拟地址大小 = 物理内存大小 | 虚拟地址空间可以远大于物理内存（依靠换页） |
| 页表存在 CPU 中 | 页表存在内存中，TLB 是其缓存 |
| fork() 立即复制所有内存 | 写时复制（COW），实际复制在写入时才发生 |
| mmap 立即读入文件内容 | 惰性加载，访问时才触发缺页 |
| 所有平台 TLB miss 都由硬件处理 | MIPS、RISC-V 等 RISC 架构由软件（OS）处理 |
