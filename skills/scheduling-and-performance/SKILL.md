当学生询问 CPU 调度算法、调度指标、响应时间分析、Amdahl 定律、性能瓶颈分析、实时调度等问题时，先让学生说出自己的理解，再用以下框架引导。

## 调度的核心目标

不同场景下调度目标不同——先让学生明确场景：

| 场景 | 主要目标 | 典型系统 |
|------|---------|---------|
| 批处理（Batch） | 吞吐量（Throughput），周转时间（Turnaround） | HPC 集群 |
| 交互式（Interactive） | 响应时间（Response Time），公平性 | 桌面 OS |
| 实时（Real-time） | 截止时间（Deadline），可预测性 | 嵌入式、工控 |
| 服务器（Server） | 吞吐量，尾延迟（Tail Latency） | Web 服务 |

提问：「你在分析的场景，最重要的调度指标是什么？吞吐量和响应时间有时候会冲突，你希望如何权衡？」

## 核心调度算法

### FCFS（First-Come, First-Served）
- 非抢占，按到达顺序执行
- **护航效应（Convoy Effect）**：一个长任务后面跟着大量短任务，平均等待时间极长
- 提问：「如果一个 100ms 的任务排在 99 个 1ms 任务前面，平均等待时间是多少？」

### SJF / STCF（Shortest Job First / Shortest Time to Completion First）
- SJF 非抢占，STCF 抢占：新任务到达时，若剩余时间更短则抢占当前任务
- **平均周转时间最优**（但需要知道执行时间，实际中需预测）
- 可能导致长任务饥饿（Starvation）

### Round Robin（时间片轮转）
- 每个任务运行固定时间片（quantum）后被抢占，放回队尾
- 时间片大小权衡：太小 → 上下文切换开销大；太大 → 退化为 FCFS
- 对所有任务公平，**响应时间好**，但周转时间比 SJF 差

### 多级反馈队列（MLFQ）
- 多个优先级队列，新任务默认高优先级，用完时间片则降级
- **自适应调度**：短任务自然在高优先级队列完成，长任务降到低优先级
- 防饥饿：定期将所有任务提升到最高优先级（Priority Boost）
- Linux 的 CFS（Completely Fair Scheduler）是更复杂的公平调度实现

### 实时调度
- **EDF（Earliest Deadline First）**：总是选截止时间最早的任务 → 理论上最优（可利用率 ≤ 100%）
- **Rate Monotonic**：优先级与任务频率正比 → 静态优先级，简单可分析
- 提问：「EDF 和 RM 的可利用率上限分别是多少？」

## 排队论与性能分析

**关键公式（M/M/1 队列）**：
- 服务率 = μ，到达率 = λ，利用率 = **ρ = λ/μ**
- 平均队列长度 = **ρ / (1 - ρ)**（利用率接近 1 时，队列爆炸）
- 平均响应时间 = **(1/μ) / (1 - ρ)**

**实践含义**：系统利用率不能接近 100%，否则延迟急剧上升
- 设计目标：利用率 ≤ 70-80%
- 提问：「如果一台服务器 CPU 利用率 90%，平均响应时间是利用率 50% 时的多少倍？」（答：约5倍）

## Amdahl 定律（并行加速极限）

**公式**：加速比 = 1 / (s + (1-s)/n)
- s = 串行部分比例，n = 处理器数量
- 结论：若串行部分占 10%，无论多少核心，加速比不超过 10×

**推论**：
- 优化热点：先找最慢的部分（用 profiler），而不是随机优化
- Lampson：**"先设计，再编码调试，再测量，最后才优化"**
- 提问：「你知道程序的瓶颈在哪吗？用 perf/gprof 测过吗？」

## 动态 vs. 静态资源分配

| | 静态分配 | 动态分配 |
|-|---------|---------|
| 原理 | 预先为每个任务分配固定资源 | 运行时按需分配/回收 |
| 优点 | 可预测，无竞争 | 资源利用率高，灵活 |
| 缺点 | 资源浪费（分配了但没用） | 竞争，可能死锁或饥饿 |
| OS 体现 | 实时系统预留 CPU 时间，IOMMU 静态映射 | 虚拟内存、动态 NUMA 分配 |

## 上下文切换成本

上下文切换不仅仅是保存/恢复寄存器：
- TLB flush（切换进程时，TLB 中的映射失效）
- Cache 污染（新进程工作集与旧进程不同）
- 分支预测状态（BTB 被污染）

实际测量：进程切换约 1-10μs，线程切换约 1-3μs（同进程内共享地址空间，TLB 不刷新）

## 性能分析方法论

Lampson 的性能分析步骤：
1. **测量先行**：用 `perf`, `strace`, `vmstat`, `iostat` 找实际瓶颈
2. **找瓶颈**：哪个资源（CPU/内存/磁盘/网络）利用率最高？
3. **算 Amdahl**：优化该瓶颈能带来多大整体提升？
4. **快路径优化**：让 90% 的常见情况走最快路径

**尾延迟（Tail Latency）问题**：
- P99 延迟往往比平均延迟高 10-100 倍
- 原因：GC 停顿、锁竞争、内核调度抖动
- 对策：请求对冲（Hedged Requests）、超时+重试

## 常见学生误区

| 误区 | 正确理解 |
|------|---------|
| SJF 总是最好的调度算法 | 需要知道任务长度；会导致长任务饥饿；实际中只能预测 |
| CPU 利用率越高越好 | 利用率 > 80% 时排队延迟急剧上升，应留有余量 |
| 多核就能线性加速 | Amdahl 定律限制，串行部分是上限 |
| 优先级调度 = 实时调度 | 普通优先级调度不保证截止时间；真正的实时需要 EDF/RM |
